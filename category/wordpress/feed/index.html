I was looking for a plugin that created an accessible accordion for Gutenberg editor considering its a common request for UI. I must have spent hours, days looking for 1, and I found a few…however, they all used the <details> and <summary> tag, which if you are in the know – you know it’s not an accordion .

0
I know that I have left the blue focus style, and some interesting style choices, but these are free to be modified at your discretion

I searched and searched and each find was in my view not sufficient, even the WordPress team are talking about it, and they even suggest a <details> component. I think the Details HTML should be used, but not for the purpose of what it’s illustrating. Thankfully, they are launching their own as part of the Gutenberg editor which will be interesting to see.

If you can’t find it, develop your own.

I decided that I’d take the plunge and develop an actual accordion toggle, that uses the WCAG accordion example, so you know it’s accessible and to a good standard already.

Feel free to use it, fork it, clone it, but please don’t just copy it and say it’s your own work – contribute to it :)!

]]> https://www.russforster.co.uk/accessible-accordion-for-gutenberg/feed/ 0 I don’t know about Gutenberg. https://www.russforster.co.uk/i-dont-know-about-gutenberg/ https://www.russforster.co.uk/i-dont-know-about-gutenberg/#respond Thu, 07 May 2020 16:05:57 +0000 http://localhost:10012/?p=136 I’ll start this off by saying, hats off to the WordPress dev team. They’ve done a fantastic job in improving the interface and post creation process to give more control back to the user – that’s great. But what does that mean for devs like me who need to develop themes that need safety nets for design?

Well, this theme i’ve built entirely on Gutenberg, it feels pretty flexible in that sense – but I know what i’m doing. I wouldn’t expect Joe Blogs to understand or even comprehend the layout aspect half the time. But with Gutenberg, they sort of have to – unless you use Templates and restrict them on what blocks they can use but that defeats the point, doesn’t it?

Honestly, it felt half the time I was battling the editor to move and group blocks in the way I want, even the custom blocks i’ve developed (tutorial coming soon) for this theme just didn’t seem to want to work.

But whilst writing this article, it actually feels pretty nice, and looks cool also. Using only basic blocks though it would feel like that. Introducing UI block interactions is a different story.

Would I choose Gutenberg again to develop websites for customers? Not sure yet, it’s a judgement call on how tech savy they are. If they aren’t then I would use my beloved Advanced Custom Fields PRO plugin that I’ve been using for years – you can do some really neat and intelligent stuff with it to help offload the burden to end users who “just want a flashy site”.

Accepting that Gutenberg is here to stay is hard for me, I like being able to develop bespoke systems with the use of ACF Pro and various other tools/custom PHP. It’s quite a mind shift personally and i’ve sort of been relearning how to do WordPress development whilst using Gutenberg.

Is it over engineered? Possibly. Is it perfect? No. Definitely not. Is it a step in the right direction – yes.
